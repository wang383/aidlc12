## 问题陈述

产品经理（PM）在需求评审阶段，通常只能提供文字描述或简单线框图。研发和设计师需要反复沟通才能对齐理解，导致：
- 需求到定稿周期长（多轮沟通）
- 设计师花大量时间做基础"搭积木"工作（标准列表页、表单页等）
- 研发对需求理解偏差导致返工

**核心痛点**：PM 缺乏一个快速将文字需求转化为"所见即所得"交互式 DEMO 的工具，导致产研设计三方对齐效率低下。

---

## 前提（Premises）

P1. 公司前端标准组件库直接使用 Element Plus（Vue 3），无二次封装。（来源：用户 Q1 回答 C）

P2. PM 是第一使用者，直接在 Kiro 中输入需求描述触发生成。设计师和研发是 DEMO 的消费者。（来源：用户 Q2 回答 A）

P3. 企业应用 80% 的基础场景可归纳为 4 种标准页面类型：列表页、表单页、详情页、仪表盘页。（来源：研究报告 Finding 2，vue-element-admin 87k+ stars 验证）

P4. 公司网络可直接访问 unpkg.com / cdn.jsdelivr.net 等公共 CDN。（来源：用户 Q4 回答 A）

P5. 模板填充式（方案 A）是 V1 的正确起点——用预置 HTML 骨架 + LLM 填充，验证价值后 V2 可演进到 JSON IR 架构。（来源：用户方案选择 A）

## 目标用户

**主要用户**：产品经理（PM）——在 Kiro 中输入自然语言需求描述，触发 DEMO 生成。不需要了解 Vue、Element Plus 或任何前端技术。

**次要用户**：
- 设计师——拿到 DEMO 后跳过基础搭建，专注 20% 的核心审美判断和最终 UI 方案决策
- 前端研发——拿到 DEMO 后快速确认需求理解，对齐技术方案

## 需求

### 核心功能

**R1. 自然语言需求解析**
Skill 接受 PM 输入的自然语言需求描述（中文），从中提取实体名称、字段列表、字段类型、页面类型和操作动作。

验收示例（AE1）：输入"用户管理模块，包含姓名、邮箱、角色（管理员/普通用户）、状态（启用/禁用）、创建时间，支持搜索、分页、新增和编辑"，Skill 正确识别出实体"用户"、5 个字段及其类型、页面类型"列表页"、操作"新增/编辑"。

**R2. 独立 HTML 文件输出**
生成单个 .html 文件，通过 CDN 加载 Vue 3 + Element Plus + @element-plus/icons-vue，无需任何构建工具，PM 双击即可在浏览器中打开查看。

CDN 引用（版本锁定）：
```
Vue 3: //unpkg.com/vue@3.4.38/dist/vue.global.prod.js
Element Plus CSS: //unpkg.com/element-plus@2.9.1/dist/index.css
Element Plus JS: //unpkg.com/element-plus@2.9.1
Icons: //unpkg.com/@element-plus/icons-vue@2.3.1
```

**R3. 四种标准页面模板**
V1 支持 4 种页面类型，每种有明确的组件构成：

| 页面类型 | 核心组件 | 触发关键词 |
|---------|---------|-----------|
| 列表页 | el-table + el-pagination + el-form(inline 搜索) + el-dialog(CRUD 弹窗) + el-button(操作) | 列表、管理、查看全部、搜索、筛选 |
| 表单页 | el-form + el-form-item + 验证规则 + el-button(提交/取消) | 创建、新增、编辑、提交、配置、注册 |
| 详情页 | el-descriptions + el-card + el-tabs(可选) | 查看、详情、档案、展示 |
| 仪表盘 | el-row/el-col + el-card(统计卡片) + el-progress + el-statistic | 概览、统计、分析、指标、监控 |

**V1 范围**：每次调用生成单个页面。列表页内嵌 CRUD 弹窗（el-dialog + el-form）解决新建/编辑场景。不支持多页面路由导航。

**R4. 字段到组件的自动映射**
根据字段描述自动推断 Element Plus 组件：

| 字段特征 | 组件 | 决策规则 |
|---------|------|---------|
| 短文本（名称、标题、编号） | el-input | 默认，无特殊标记 |
| 长文本（描述、备注、内容） | el-input type="textarea" | 字段名含"描述/备注/内容/说明"，或明确标注"多行" |
| 邮箱 | el-input | 字段名含"邮箱/email" |
| 数字（价格、数量、金额） | el-input-number | 字段名含"价格/数量/金额/数/次数" |
| 日期 | el-date-picker | 字段名含"日期/时间/创建时间/更新时间/截止" |
| 枚举（≤5 个选项） | el-radio-group | 字段明确列出选项且 ≤5 个 |
| 枚举（>5 个选项） | el-select | 字段明确列出选项且 >5 个，或字段名含"类型/分类/类别" |
| 布尔 | el-switch | 字段名含"是否/启用/禁用/开关" |
| 文件/图片 | el-upload | 字段名含"图片/头像/文件/附件/上传" |
| 多选标签 | el-select multiple | 字段名含"标签/多选" |

**表格列渲染规则**：
- 枚举状态字段 → el-tag（不同值不同颜色：成功=success，警告=warning，危险=danger）
- 日期字段 → 格式化显示（YYYY-MM-DD）
- 布尔字段 → el-tag（是=success，否=info）
- 其他 → 纯文本

**R5. 国际化支持（可选，默认关闭）**
V1 默认生成硬编码中文字符串。可通过在 prompt 中指定"需要国际化"启用 i18n 模式，此时：
- 加载 vue-i18n v11 via CDN
- 所有用户可见文本使用 `$t('key')` 替代
- 内联 zh-CN 和 en 两套 locale 数据
- Element Plus locale 同步切换
- i18n key 命名规范：`{feature}.{elementType}.{name}`（如 `userManagement.columns.name`）

**R6. 图标自动匹配**
根据业务概念自动匹配 Element Plus 内置图标：

| 场景 | 图标 |
|------|------|
| 新增按钮 | Plus |
| 编辑按钮 | Edit |
| 删除按钮 | Delete |
| 搜索按钮 | Search |
| 导出按钮 | Download |
| 刷新按钮 | Refresh |
| 查看详情 | View |

**R7. 模拟数据生成**
生成 10-20 行上下文相关的模拟数据，内联为 JSON 数组：
- 姓名字段 → 中文姓名（如"张三"、"李四"）
- 手机字段 → 合法格式手机号
- 邮箱字段 → 合法格式邮箱
- 日期字段 → 近 30 天内的合理日期
- 枚举字段 → 从选项中随机分布
- 数字字段 → 合理范围内的数值

### 质量保障

**R8. 生成后结构校验**
生成 HTML 后执行基本校验：
- Vue app 挂载点 `#app` 存在
- 所有引用的 Element Plus 组件标签在已知组件列表中
- 无未闭合的 HTML 标签
- `<script>` 块中无明显语法错误（括号匹配）

校验失败时：向用户报告具体问题，尝试自动修复一次。修复失败则输出带问题标注的 HTML 并提示 PM 手动调整或重新描述需求。

**R9. 对话式迭代修改**
支持 PM 通过后续对话修改已生成的 DEMO。V1 支持的修改操作：
- 增加/删除字段
- 修改字段的组件类型（如 input → select）
- 修改模拟数据
- 修改页面标题和标签文字
- 调整表格列的显示/隐藏

V1 不支持：更换页面类型、添加新页面、结构性布局变更。

### 扩展性

**R10. Steering 文件定制接口**
通过 `.kiro/steering/demo-generator.md` 支持公司级定制：
- **图标映射覆盖**：自定义业务概念到图标的映射
- **组件偏好覆盖**：如"所有枚举字段统一用 el-select，不用 el-radio"
- **默认主题色**：通过 CSS 变量覆盖 Element Plus 主色调
- **i18n 默认行为**：设置默认开启/关闭

格式：Markdown 文件，使用特定 section 标题标识覆盖类型。

## 方案选择

### 考虑的方案

**方案 A：模板填充式（选中）**
4 套预置 HTML 模板骨架，LLM 从需求中提取信息直接填充。
- 优点：最快、最可控、失败面最小
- 缺点：灵活性低，非标布局难支持
- 标签：Build new

**方案 B：JSON IR + 代码生成**
两阶段：需求 → JSON 中间层 → HTML。
- 优点：灵活、可扩展、可定制
- 缺点：复杂度高、token 消耗大、JSON 解析可能失败
- 保留为 V2 演进方向

**方案 C：可视化需求规格（反转思路）**
不生成代码，生成结构化 UI 规格文档。
- 优点：避开代码质量风险
- 缺点：失去"所见即所得"核心卖点
- 作为降级方案保留

### 推荐方案

**方案 A**。理由：PM 是第一使用者，核心指标是速度（< 60 秒生成）。模板填充最快最可控。4 种模板的 Element Plus 组件构成已在研究中完全明确，模板质量可在 Skill 开发阶段打磨到位。

**翻转条件**：如果 V1 上线后发现模板填充的输出质量无法满足 60%+ 的场景（即 PM 频繁需要大幅手动修改），则切换到方案 B 的 JSON IR 架构。

## 每种页面模板的组件规格

### 列表页模板规格

```
┌─────────────────────────────────────────────┐
│ 页面标题                          [新增] [导出] │
├─────────────────────────────────────────────┤
│ 搜索栏（el-form inline）                      │
│ [字段1 input] [字段2 select] [搜索] [重置]     │
├─────────────────────────────────────────────┤
│ 数据表格（el-table）                           │
│ ┌──────┬──────┬──────┬──────┬────────┐      │
│ │ 字段1 │ 字段2 │ 字段3 │ 状态  │  操作   │      │
│ ├──────┼──────┼──────┼──────┼────────┤      │
│ │ ...  │ ...  │ ...  │ tag  │ 编辑 删除│      │
│ └──────┴──────┴──────┴──────┴────────┘      │
├─────────────────────────────────────────────┤
│ 分页（el-pagination）                         │
│              < 1 2 3 ... 10 >                │
└─────────────────────────────────────────────┘

CRUD 弹窗（el-dialog + el-form）：
┌─────────────────────────┐
│ 新增/编辑 [X]            │
│ ┌─────────────────────┐ │
│ │ 字段1: [input]      │ │
│ │ 字段2: [select]     │ │
│ │ 字段3: [date-picker]│ │
│ └─────────────────────┘ │
│         [取消] [确定]    │
└─────────────────────────┘
```

注入点：页面标题、搜索字段列表（searchable=true 的字段）、表格列（showInTable=true 的字段）、表单字段（所有字段）、操作按钮、模拟数据。

### 表单页模板规格

```
┌─────────────────────────────────────────────┐
│ 页面标题                                      │
├─────────────────────────────────────────────┤
│ el-form (label-width: 120px)                │
│ ┌─────────────────────────────────────────┐ │
│ │ 字段1标签: [对应组件]                      │ │
│ │ 字段2标签: [对应组件]                      │ │
│ │ 字段3标签: [对应组件]                      │ │
│ │ ...                                     │ │
│ └─────────────────────────────────────────┘ │
│                        [取消] [提交]         │
└─────────────────────────────────────────────┘
```

注入点：页面标题、字段列表（按顺序）、每个字段的组件类型和验证规则、按钮文字。

### 详情页模板规格

```
┌─────────────────────────────────────────────┐
│ 页面标题                          [编辑] [返回] │
├─────────────────────────────────────────────┤
│ el-descriptions (border)                    │
│ ┌──────────┬────────────────────────┐       │
│ │ 字段1标签 │ 字段1值                 │       │
│ │ 字段2标签 │ 字段2值                 │       │
│ │ 字段3标签 │ 字段3值                 │       │
│ └──────────┴────────────────────────┘       │
└─────────────────────────────────────────────┘
```

注入点：页面标题、字段列表、字段值（取模拟数据第一行）、操作按钮。

### 仪表盘模板规格

```
┌─────────────────────────────────────────────┐
│ 页面标题                                      │
├─────────────────────────────────────────────┤
│ el-row :gutter="20"                         │
│ ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│ │ 统计卡片1  │ │ 统计卡片2  │ │ 统计卡片3  │     │
│ │ 指标名     │ │ 指标名     │ │ 指标名     │     │
│ │ 数值       │ │ 数值       │ │ 数值       │     │
│ └──────────┘ └──────────┘ └──────────┘     │
├─────────────────────────────────────────────┤
│ el-card                                     │
│ ┌─────────────────────────────────────────┐ │
│ │ 数据表格或列表（简化版）                     │ │
│ └─────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
```

注入点：页面标题、统计指标列表（名称+数值+图标）、下方内容区（可选表格或列表）。

## 范围边界

### V1 包含（IN）
- 4 种页面类型的单页面生成
- 列表页内嵌 CRUD 弹窗
- 字段到 Element Plus 组件的自动映射
- 中文模拟数据
- Element Plus 内置图标匹配
- 对话式迭代修改（增删字段、改组件类型、改文字）
- i18n 支持（可选，默认关闭）
- Steering 文件定制接口

### V1 不包含（OUT）
- 多页面路由导航
- 真实 API 对接
- 移动端响应式布局（V1 目标桌面端 1280px+）
- 图表/数据可视化（ECharts 等）
- 截图/图片输入
- 导出到 Figma 或其他设计工具
- JSON IR 中间层（保留给 V2）
- 自定义主题色（除非通过 steering 文件的 CSS 变量覆盖）

---

## 未解决问题

### 延迟到实施阶段（Deferred to Implementation）
- 公司的全局 Icon 规范和国际化规范的具体文档内容——V1 使用 Element Plus 默认，通过 steering 文件预留定制接口
- CDN 版本更新策略——V1 锁定特定版本，后续作为维护任务处理
- 生成的 HTML 文件大小上限——预计单文件 50-200KB，需实测确认

### 审查者关注点（Reviewer Concerns）
- 模板填充式（方案 A）在面对高度非标需求时可能输出质量不足——通过 R8 的校验机制和 R9 的迭代修改缓解，V2 可切换到 JSON IR 架构
- 研究报告中的 Vue 2 + Element UI 推荐已被用户确认的 Vue 3 + Element Plus 取代——PRD 以用户确认为准，忽略研究报告中的 Vue 2 相关内容
